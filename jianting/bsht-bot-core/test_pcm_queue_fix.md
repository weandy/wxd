# PCM 队列溢出修复测试指南

> **日期**: 2026-03-04
> **问题**: 音频广播时大量丢帧（6 秒音频只收到 0.3 秒）
> **修复**: 实现队列流量控制机制

---

## 问题回顾

### 症状
```
[TX_WEB] PCM 队列已满，丢弃帧 (×100+)
[播放] 播放进度: 6.0s, 300 帧
最终 App 端只收到 0.3 秒音频
```

### 根本原因
- 队列容量：50 帧 = 1 秒
- 主线程喂帧速度：每 20ms 一帧（无延迟）
- 后台编码速度：Opus + RTP 处理较慢
- **结果**：队列瞬间填满，后续帧全部丢弃

---

## 修复方案

### 1. bot_server.py - 队列流量控制
```python
# ✅ 关键修复：队列满时等待
while True:
    try:
        self.listener.feed_web_pcm(pcm_bytes)
        break  # 成功，继续
    except queue.Full:
        time.sleep(0.010)  # 等待 10ms
        if not self.listener.is_transmitting:
            break
        continue  # 重试
```

### 2. bsht_client.py - 重新抛出异常
```python
# ✅ feed_web_pcm 不再捕获 queue.Full
# 让调用者处理背压
self._tx_web_pcm_queue.put_nowait(frame_data)  # 队列满时抛出异常
```

## 工作原理

### 背压机制 (Backpressure)
```
主线程                    队列 (50 帧)                  编码线程
  |                         |                            |
  |--帧1--> [成功] -------> |                            |
  |--帧2--> [成功] -------> |<-----处理------ 空出槽位   |
  |                         |                            |
  |--帧51> [队列满] -----> |✗ 满!                      |
  |      [等待 10ms]        |                            |
  |--帧51> [重试成功] ----> |<-----处理------ 空出槽位   |
  |--帧52--> [成功] -------> |                            |
```

### 关键参数
- **队列容量**: 50 帧 = 1 秒
- **等待时间**: 10ms（平衡响应性和吞吐量）
- **总等待**: 6 秒音频可能额外延迟 0.5-1 秒

## 故障排查

### 如果仍然丢帧

1. **检查队列容量**:
```python
# bsht_client.py 中
self._tx_web_pcm_queue = queue.Queue(maxsize=50)  # 可以增加到 100
```

2. **增加等待时间**:
```python
# bot_server.py 中
time.sleep(0.020)  # 改为 20ms
```

3. **检查系统负载**:
```bash
# CPU 使用率过高可能导致编码慢
top  # Linux
taskmgr  # Windows
```

### 如果播放卡顿

可能原因：等待时间太长导致播放不流畅

**解决方案**:
- 减少等待时间到 5ms
- 增加队列容量到 100 帧
